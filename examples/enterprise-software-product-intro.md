# 示例：企业软件产品介绍

用户输入：

```text
我们要给一家大型制造企业介绍公司的供应链管理软件，希望让对方相信这个产品能帮助他们降低库存压力、提升交付可视化，并愿意安排一次后续方案评审。汇报时间 30 分钟以内。
```

建议输出方向：

## 1. 输入收敛

| 类型 | 内容 | 对框架的影响 |
| --- | --- | --- |
| 已确认事实 | 听众是大型制造企业；产品是供应链管理软件；目标是争取后续方案评审 | 使用客户销售/解决方案提案结构 |
| 合理假设 | 客户关心库存、交付、系统集成和落地风险 | 先讲业务痛点，再讲产品能力 |
| 高影响未知 | 听众角色、现有系统、可公开案例、ROI 证据 | 证明页必须保留占位，不编造数据 |

## 2. 推荐框架

1. **制造企业的供应链压力已经从“缺数据”变成“缺可行动判断”**
   - 讲库存、交付、协同和异常响应的业务背景。

2. **当前方式的核心成本不是系统数量，而是计划与执行不可视**
   - 展示典型断点：库存积压、交付延迟、跨部门信息差。

3. **我们的软件把供应链从事后统计变成过程可视和提前预警**
   - 用一张工作流说明产品核心价值。

4. **三个场景最适合先验证价值**
   - 库存预警、交付追踪、异常协同。

5. **已有证明与待验证假设要分开看**
   - 放客户案例、试点结果或占位证明。

6. **建议下一步做一次 90 分钟方案评审，而不是直接采购决策**
   - 降低客户压力，明确评审输入和输出。

## 3. 需要补充的证据

- 客户当前供应链系统和痛点
- 过往案例或试点数据
- 集成方式、上线周期、风险控制
- 方案评审需要客户提供的材料
