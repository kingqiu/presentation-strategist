# Classic Frameworks

Use frameworks as tools, not identities. Select based on communication task, audience resistance, evidence strength, and field.

## Pyramid Principle

Best for:

- executive updates
- recommendations
- decision decks
- consulting-style business arguments

Pattern:

```text
answer -> supporting reasons -> evidence -> implications / ask
```

Use when the audience needs fast judgment. Avoid long suspense.

## SCQA

Best for:

- opening a business problem
- making context and tension clear
- turning background into a decision question

Pattern:

```text
Situation -> Complication -> Question -> Answer
```

Use as an opening device, not necessarily the whole deck.

## MECE Decomposition

Best for:

- root-cause analysis
- option comparison
- market segmentation
- risk categories

Rule:

```text
Break the issue into non-overlapping, collectively useful parts.
```

Do not make MECE more important than audience usefulness.

## Changed-World Strategic Narrative

Best for:

- B2B sales
- category creation
- strategy change
- product adoption

Pattern:

```text
world changed -> old approach breaks -> new winners/losers -> promised land -> product/path -> proof
```

Use when the audience must first believe that staying still is costly.

## Problem / Cost / Solution / Proof / Action

Best for:

- proposals
- product introductions
- customer pitches

Pattern:

```text
problem -> cost of inaction -> solution -> proof -> next action
```

Use when the pain is known but urgency or solution confidence is weak.

## Current State / Desired State / Bridge

Best for:

- transformation plans
- internal alignment
- change management

Pattern:

```text
current state -> desired state -> gap -> bridge -> operating implications
```

Use when the audience needs to see the path, not just the destination.

## Hero / Obstacle / Success

Best for:

- motivational or launch narratives
- career pitches
- customer transformation stories

Pattern:

```text
protagonist -> goal -> obstacle -> guide/path -> proof -> success state
```

Usually make the audience the protagonist. The speaker, product, or company should be the guide or path.

Do not force this onto technical reviews, risk briefings, audits, or crisis communication.

## What Is / What Could Be

Best for:

- keynotes
- product launches
- change narratives

Pattern:

```text
what is -> what could be -> what is -> what could be -> call to action
```

Use contrast to create emotional and cognitive movement.

## Ethos / Pathos / Logos

Use as a persuasion balance check:

- Ethos: why should the audience trust this speaker/source?
- Pathos: why should the audience care?
- Logos: why does the argument hold?

Do not let emotion outrun proof.

## Amazon PR/FAQ

Best for:

- product ideas
- innovation proposals
- customer-backward thinking

Pattern:

```text
future press release -> customer problem -> promised customer outcome -> FAQ objections -> constraints
```

Recommend before slides when the product idea is unclear or customer value is unproven.

## Six-Page Narrative Memo

Best for:

- complex decisions
- high-stakes strategy
- technical or financial nuance
- asynchronous executive review

Use when slides would compress evidence too much.

Pattern:

```text
context -> thesis -> evidence -> alternatives -> risks -> recommendation -> decision needed
```

## Risk Briefing

Best for:

- crisis
- technical risk
- safety, legal, financial, or delivery risk

Pattern:

```text
facts -> impact -> uncertainty -> options -> recommendation -> decision needed -> monitoring
```

Make uncertainty visible. Do not use narrative polish to soften material risk.

## Teaching Ladder

Best for:

- training
- knowledge sharing
- onboarding

Pattern:

```text
learner gap -> concept -> example -> practice -> feedback -> transfer
```

Every teaching deck should end with a behavior the learner can perform.

## Option Comparison

Best for:

- technical reviews
- strategy decisions
- vendor or architecture choices

Pattern:

```text
decision criteria -> options -> tradeoffs -> recommendation -> validation / fallback
```

Do not present options before criteria.
